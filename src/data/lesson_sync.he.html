Sync: Semaphores, Shared Memory & Critical Sections

סנכרון מונע מצב מירוץ (race condition) שבו התוצאה תלויה בסדר הביצוע. הבחינה אוהבת מונחים מדויקים והתנהגות של קריאות System V: sem_op=-1 הוא wait, +1 הוא signal, וצריך לדעת בעל-פה את 3 התנאים של הקטע הקריטי. כתבו את המונח באנגלית גם אם אתם מסבירים בעברית.

Solving procedure — semop scenario

  1. כתבו את הערך ההתחלתי של כל סמפור (לאחר SETVAL).
  2. עבור sem_op=-1: אם value>0 הפחיתו; אם value=0 התהליך נחסם (sleep/WAITING).
  3. עבור sem_op=+1: הגדילו והעירו ממתין אחד.
  4. עבור sem_op=0: המשיכו אם value=0, אחרת היחסמו עד שיגיע ל-0.
  5. אם nsops>1: כל הפעולות מצליחות יחד או שהקריאה כולה נחסמת (פילוסופים = 2 מקלות בו-זמנית).
Start: S=1
wait(S)  : sem_op=-1 → S=0, RUNNING
wait(S)  : sem_op=-1 → S=0, process SLEEPS (was 0)
signal(S): sem_op=+1 → S=0, waiter woken → RUNNING
Close-but-precise traps:
  • sem_op=-1 על סמפור 0 → נחסם/ישן (לא שגיאה, לא -1). תשובת 2024-B Q10 = sleep.
  • sem_op=-1 הוא wait/lock; +1 הוא signal/unlock — לעולם לא הפוך.
  • הגדרת sops.sem_op=-1 לבדה אינה עושה דבר — היא פועלת רק כשקוראים ל-semop().
  • לאחר IPC_RMID המערך נעלם; semop מאוחר יותר מחזיר -1.
  • 3 התנאים הם Mutual Exclusion, Progress, Bounded Waiting. הרעבה (starvation) היא תוצאה, ולא תנאי.
  • Bakery ללא מערך ה-choosing[] שובר את Mutual Exclusion (שניים בוחרים אותו מספר, שניהם נכנסים).
  • Spinlock סובל מ-busy waiting + הרעבה; הגרעין משתמש בו כי הוא אינו יכול לישון.
  • בסדר היצרן-צרכן יש חשיבות: wait(empty) לפני wait(mutex), אחרת קיפאון.
  • semctl לבדו מבצע SETVAL, GETVAL וגם IPC_RMID — אותה קריאה, cmd שונה.
  • לזיכרון משותף אין סנכרון מובנה — תמיד הגנו עליו בסמפור. גודל shmget מעוגל כלפי מעלה לדפים של 4K.
  • תנאי הקיפאון של Coffman (4): Mutual Exclusion, Hold-and-Wait, No Preemption, Circular Wait. שבירת אחד → אין קיפאון.
  • TestAndSet היא פקודת CPU אטומית אחת (קריאת ערך ישן + קביעת true); אל תבלבלו עם שני צעדים נפרדים.
  • שכחת IPC_RMID / ipcrm משאירה אובייקטי IPC עד אתחול מחדש — נחשב לשגוי בתשובות קוד.
