Systems Concepts: the OS Vocabulary

This is the smallest, easiest topic — pure memorization of a handful of definitions. It is the big-picture vocabulary of operating systems: how multiple CPUs share resources, what an OS-with-many-processes is called, how we measure performance, what device drivers do, and the real-time categories. There is no math and no diagrams. Each card simply maps one description to one term, so your only job is to match the description to its term.

Because every card is a definition-to-term match, a short decision checklist is all you need to answer fast and correctly.

  1. Coupling: the component shared across CPUs in tightly coupled but separate per CPU in loosely coupled is the main memory (RAM).
  2. Real-time: the words most / not always point to soft real-time; the word always points to hard real-time.
  3. Performance: throughput is jobs finished per unit time — a system metric, not a per-process clock.
  4. Hardware access: a device driver mediates between hardware and applications, giving the OS a uniform way to talk to each device.

The traps that cost points:
  • Tightly coupled = shared RAM; loosely coupled = private memory per CPU. Coupling is about memory sharing, not CPU count.
  • Multiprogramming doesn't need multiple CPUs: it means many processes resident and alternated — one CPU is enough.
  • Throughput vs turnaround: throughput = jobs finished per time (a system rate); turnaround/waiting time = per-process clocks from the Scheduling topic.
  • Device driver vs system call: the driver is kernel code for one device; a system call is the userspace entry into the kernel. Different layer.
  • Soft = most of the time, hard = always: "most (not all)" means soft real-time; "always / guaranteed" means hard real-time.
