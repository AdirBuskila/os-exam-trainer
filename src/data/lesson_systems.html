No heavy computation lives in this section, but the timer-interrupt reasoning is exam-recurring. Use this checklist to answer the "who stops it / why" archetype.

  1. Spot the symptom: a process runs forever (e.g. while(1)) and holds the CPU.
  2. Name the stopper: the timer interrupt forces RUNNING → READY (preemption).
  3. Name the source: the hardware Timer / PIT, not the OS or CPU instruction.
  4. State the purpose: without it, a non-yielding process would hold the CPU forever.
Q: "A process is stuck in an infinite loop. What stops it?"  -> Timer interrupt
Q: "Who generates that interrupt?"                          -> Hardware Timer (PIT)
Q: "System meeting its deadline 99% of the time?"           -> Soft real-time
Q: "No master/slave relation between CPUs?"                 -> Symmetric multiprocessing
Close-but-precise pitfalls:
  • Hard vs Soft real-time: "always" = Hard; "most of the time / 99%" = Soft. The word "always/must" is the giveaway.
  • Symmetric vs Asymmetric: "no master/slave" / "all CPUs equal" = Symmetric. "Master handles kernel, slaves don't" = Asymmetric. Don't swap.
  • Tightly vs Loosely coupled: shared memory + clock = tightly coupled (SMP). Separate memory + network = loosely coupled (distributed). Coupling is about memory sharing, not CPU count.
  • Multiprogramming vs Time-sharing: multiprogramming = keep CPU busy (throughput goal); time-sharing = fast switching for interactive users (response-time goal).
  • Timer interrupt is hardware, not software: the Timer/PIT generates it; the OS only handles it. It is the mechanism behind preemption.
  • DMA bypasses the CPU and writes to RAM (not to the CPU, not to cache). Spool sits on the HDD, not in cache.
  • Throughput vs Turnaround: throughput = jobs finished per unit time (a rate); turnaround = time for one job to finish. Don't confuse the benchmark metric.
  • Device driver vs system call: the driver is kernel code for one device; a system call is the userspace entry into the kernel. Different layer.
