fork() — ספירת תהליכים והדפסות

כל הנושא הוא בעיה אחת: fork() משכפל את התהליך הרץ, ושני העותקים ממשיכים מאותה שורה הבאה. בהינתן תוכנית C קצרה עם קריאת fork אחת או יותר, הבחינה מבקשת לספור כמה תהליכים קיימים ו/או כמה פעמים שורה מודפסת. שרטטו את העץ, סמנו את ערכי ההחזרה, החילו את התנאי, וספרו — כל שאלה Q9 נופלת לשיטה הזו.

  1. שרטטו את עץ התהליכים. התחילו מתהליך אחד; כל fork שענף מגיע אליו מכפיל אותו. העלים הם סך התהליכים.
  2. סמנו כל תהליך בערך ההחזרה שראה בכל fork (האב קיבל PID>0, הבן קיבל 0).
  3. החילו את התנאי. עם &&/|| ייתכן שיהיה fork מדולג (עץ קטן יותר); עם */-/+/== כל הקריאות רצות. בדקו את התנאי לכל תהליך כדי לסמן מי מדפיס.
  4. ספרו את התהליכים המסומנים (וזכרו לכלול את תהליך האב המקורי).
Plain forks: N calls reached by everyone -> 2^N processes.
  fork(); fork(); fork(); printf("x");   // 2^3 = 8 prints

fork() * fork() * fork() == 0   ->  prints 7   (2024A Q9)
  No short-circuit: all 3 forks run -> 8 processes.
  Product != 0 only for the all-parent process (3 positive PIDs).
  The other 8 - 1 = 7 had >=1 fork return 0 -> product == 0 -> print.

fork() - fork() == 0   ->  prints 1   (2024B Q9)
  No short-circuit: 2 forks run -> 4 processes.
  first - second == 0 only when both returns are equal: 0 - 0 = 0
  -> the all-child process only -> 1 print.

Rule of thumb:  *...==0 -> 2^N - 1 print ;  -...==0 -> 1 prints.

  • כיוון ערך ההחזרה: האב מקבל את PID של הבן (חיובי), הבן מקבל 0, כישלון נותן -1 לאב בלבד — לעולם אל תערבבו ביניהם.
  • קצר חשמלי של && / || יכול לדלג על fork, ולכן העץ קטן מ-2N הגולמי. חישוב והשוואה (* - + ==) לעולם אינם גורמים לקצר — כל הקריאות שלהם רצות.
  • קוד אחרי while(1) או exit() לעולם אינו רץ — קריאות fork שמוקמו אחריו אינן יוצרות תהליכים.
  • שני שורות-הקסם: עבור *...==0 התהליך כולו-אב הוא היחיד ששונה מאפס, ולכן 2N−1 מדפיסים; עבור -...==0 התהליך כולו-בן הוא ההתאמה היחידה, ולכן רק 1 מדפיס.
  • תמיד ספרו את האב וזכרו שהבן ממשיך אחרי שורת ה-fork (הוא אינו מתחיל מחדש מ-main) — טעות נפוצה היא לספור רק את הבנים.
