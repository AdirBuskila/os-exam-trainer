זיכרון: זיכרון וירטואלי, דפדוף ומסלול ה-Page-Fault

זיכרון וירטואלי מעניק לכל תהליך מרחב כתובות מבודד הגדול יותר מ-RAM. ה-MMU מתרגם וירטואלי→פיזי דרך טבלת דפים לכל תהליך; ה-TLB מאחסן במטמון תרגומים אחרונים. רוב הנקודות במבחן כאן הן שליפת מושג במילה אחת בתוספת חישובי מסגרת/גישה קטנים בחזקות של 2.

חישובי מסגרת / דף ו-TLB-miss

  1. הביעו את RAM וגודל הדף כחזקות של 2. #frames = RAM / page_size ← החסירו מעריכים.
  2. offset ביטים = log2(page_size); ביטים של מספר-דף = address_bits − offset bits.
  3. #pages (או #frames) = 2^(page-number bits).
  4. מרחב וירטואלי רב-רמתי = (entries per level)^levels × page_size.
  5. TLB hit = גישה אחת לזיכרון (הדף). TLB miss = levels + 1 גישות (מעבר על כל N הטבלאות, ואז הדף).
42-bit virtual address, page size 1MB = 2^20
  page-number bits = 42 - 20 = 22  =>  #frames = 2^22
3-level multilevel paging, TLB miss:
  accesses = 3 (tables) + 1 (page) = 4
מלכודות קרובות-אך-מדויקות
  • מי קובע איזו סיבית: ה-CPU/MMU קובע את הסיביות Reference ו-Dirty (אוטומטית בגישה/כתיבה). ה-OS קובע את סיבית ה-Present/Valid ומאפס את סיבית ה-reference (למשל ב-Clock).
  • TLB miss = N+1 גישות, לא N — לעולם אל תשכחו את הגישה לדף עצמו.
  • מסגרות ≠ דפים: מספר המסגרות הפיזיות נגזר מגודל ה-RAM; מספר הדפים הווירטואליים נגזר ממרחב הכתובות. אל תערבבו ביניהם.
  • דף מול Swap: דפדוף מעביר דף בודד (4KB); swap (roll out/in) מעביר תהליך שלם. ב-Linux דפים שפונו עוברים אל ה-swap partition.
  • פיצול פנימי (לא חיצוני) הוא זה של הדפדוף — בזבוז בתוך הדף האחרון; בדפדוף טהור אין פיצול חיצוני.
  • COW הוא מה שהופך את fork() לזול: דפים משותפים לקריאה בלבד ומועתקים רק כשצד אחד כותב — לא מועתקים מראש.
  • טבלה רב-רמתית חוסכת זיכרון (לא מחזיקים את כל הטבלה) אך עולה בגישות נוספות לכל תרגום.
  • אנומליית Belady (יותר מסגרות ← יותר שגיאות) שייכת ל-FIFO בלבד; LRU ו-OPT הם אלגוריתמי מחסנית ולעולם אינם סובלים ממנה.
  • ענו במונח האנגלי, ‎≤5 מילים (למשל "Page fault", "Copy-on-Write") — איות בעברית מסוכן.
