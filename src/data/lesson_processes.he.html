תהליך (process) הוא תוכנית בריצה, המיוצגת על ידי ה-PCB שלה. שננו את שמות המצבים המדויקים (5 הסטנדרטיים + Linux R/S/D/T/Z), מה החלפת הקשר (context switch) שומרת, ואת כלל הזומבי/יתום. התשובות הן מילה או שתיים — העדיפו את המונח האנגלי.

  1. נשאל שם מצב → מפו את הטריגר: syscall חוסם (read/wait/semop) → RUNNING→WAITING; סיום קוונטום / טיימר → RUNNING→READY; I/O הסתיים → WAITING→READY.
  2. נשאלת אות Linux → קריאת דיסק = D; המתנה-לבן / stdin = S; Ctrl+Z = T; סיים-לא-נאסף = Z.
  3. "היכן נשמרים האוגרים בהחלפת הקשר / בחסימה?" → תמיד ב-PCB.
  4. זומבי + יתום בו-זמנית → תשובה 0 (init מאמץ ואז מיד אוסף).
  5. מספר התהליכים אחרי k קריאות fork → 2^k; getppid()==1 → האב מת.
Process does read() from disk            -> state D (uninterruptible)
Parent does wait(), child still running  -> state S (sleeping)
Child exit(0), parent hasn't wait()ed    -> state Z (zombie)
Parent dies, child keeps running         -> orphan, getppid()==1
How many can be zombie AND orphan at once -> 0
  • החלפת הקשר ≠ החלפת מצב. החלפת מצב = user↔kernel, אותו תהליך (זול). החלפת הקשר = החלפת התהליך כולו דרך ה-PCB (יקר).
  • D מול S: D = לא ניתן להפרעה (חומרה/דיסק), לא ניתן להעירו על ידי אות; S = ניתן להפרעה (המתנה רגילה).
  • R מכסה את שניהם Running ו-Ready ב-Linux — אך פורמלית אלה שני מצבים שונים.
  • זומבי + יתום = 0, לא 1. init תמיד אוסף; יתום מאומץ באופן מיידי.
  • לזומבי עדיין יש PCB (מוחזק רק כדי למסור את קוד היציאה); אין בבעלותו משאבים אחרים.
  • קוד יציאה: 0 = הצלחה, כל ערך שונה מאפס = כישלון. exit() מרוקן את חוצצי ה-libc; _exit() לא (השתמשו ב-_exit בתהליך בן שעבר fork כדי להימנע מהדפסות כפולות).
  • printf ללא \n + fork → החוצץ מועתק → הטקסט מודפס פעמיים. תקנו עם fflush או \n.
  • אוגרים בחסימה/החלפת הקשר עוברים ל-PCB — לא ל-stack, לא ל-RAM באופן כללי.
  • getppid()==1 משמעו שהאב סיים (אומץ על ידי init), ולא שאתם init.
