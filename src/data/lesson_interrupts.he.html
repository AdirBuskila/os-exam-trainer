Interrupts, Traps & Mode Switching

ה-CPU מריץ בדרך כלל קוד משתמש במצב משתמש (user mode); רק הגרעין יכול לגעת בחומרה. שלושה אירועים מעבירים אותו למצב גרעין (kernel mode): קריאת מערכת (system call) (התהליך מבקש), פסיקה (interrupt) (חומרה, אסינכרוני), או מלכודת (trap) (קוד התהליך עצמו שוגה, סינכרוני). דע את המונח המדויק ומי מפעיל אותו.

  1. קרא את הטריגר: שגיאת תהליך → מלכודת; אירוע חומרה → פסיקה; בקשה מפורשת → קריאת מערכת.
  2. כל אחד מהשלושה גורם להחלפת מצב משתמש→גרעין (אותו תהליך, זול). רק כשהמתזמן מחליף תהליכים זוהי החלפת הקשר (context switch) (שומרת/טוענת PCB, יקרה).
  3. החומרה יורה פסיקה → ה-CPU מאתר את הוקטור פסיקות (interrupt vector) → קופץ ל-ISR המתאימה → חוזר לקוד שנקטע.
  4. סווג קריאה: דורשת את הגרעין/החומרה → קריאת מערכת; קוד משתמש בלבד → קריאת ספרייה.
  5. המתנת I/O: CPU שמסתובב בלולאה = תשאול (polling); התקן-אל-RAM ללא CPU = DMA.
while(1){}   // runaway user process
  -> TIMER INTERRUPT (hardware PIT) fires at quantum end
  -> mode switch to kernel, run timer ISR (via interrupt vector)
  -> scheduler may CONTEXT SWITCH to another process
printf("hi")  -> library call (no mode switch)
write(fd,...) -> system call (mode switch into kernel)
*null = 0     -> TRAP (SIGSEGV, the process's own fault)
  • מלכודת מול פסיקה: מלכודת היא סינכרונית מקוד התהליך עצמו; פסיקה היא אסינכרונית מהחומרה. אל תחליף ביניהן.
  • החלפת מצב ≠ החלפת הקשר: החלפת מצב = משתמש↔גרעין באותו תהליך (זול); החלפת הקשר = החלפת תהליכים, שמירה/טעינה של PCB (יקר).
  • קריאת מערכת מול קריאת ספרייה: printf/strcpy/malloc הן קריאות ספרייה; fork/read/write/semop הן קריאות מערכת (החלפת מצב). (malloc עשויה לקרוא פנימית ל-brk/mmap.)
  • DMA כותב ל-RAM, לא ל-CPU — כל מטרתו היא לעקוף את ה-CPU.
  • תשאול מבזבז CPU (שואל בלי הרף); פסיקות/DMA אינם.
  • מי עוצר לולאה אינסופית? הפסיקת שעון (timer interrupt) (נוצרת על ידי החומרה — ה-PIT/טיימר), לא מתזמן ה-OS בעצמו.
  • שלוש (רק שלוש) דרכים להיכנס למצב גרעין: קריאת מערכת, פסיקה, מלכודת/חריגה. שום דבר אחר.
  • ה-spool יושב על הדיסק הקשיח (HDD) — תור הדפסה/עבודות, לא מטמון מהיר.
