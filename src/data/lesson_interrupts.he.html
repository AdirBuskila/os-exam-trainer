פסיקות ומצב ה-CPU

הנושא הזה עונה על שני צרכים של ה-CPU. ראשית, הוא חייב להגיב לאירועים מהעולם החיצוני (לחיצת מקש, סיום פעולת דיסק, תקתוק השעון) בלי לתשאל בלי הרף — מנגנון התגובה הזה הוא הפסיקה. שנית, הוא חייב להגן על עצמו כך שתוכנית משתמש שגויה או זדונית לא תוכל לגעת בחומרה או בזיכרון של תוכניות אחרות — מנגנון ההגנה הזה הוא ההפרדה בין מצב גרעין למצב משתמש. כמעט כל כרטיס בחינה כאן הוא שליפה טהורה של המונח הנכון ושל מי שמפעיל אותו.

כניסה אל הגרעיןSystem callקריאת מערכתמלכודת שתהליך משתמש בה כדי לבקש מהמערכת שירות מיוחס
  1. סווג את הטריגר: שגיאת תהליך (div/0, גישה ל-NULL) → מלכודת; אירוע חומרה חיצוני (מקלדת, דיסק, שעון) → פסיקה; בקשה מפורשת לשירות מערכת → קריאת מערכת.
  2. כשפסיקה נורית, ה-CPU מאתר את כתובת שגרת הטיפול בוקטור פסיקות, ואז קופץ ל-ISR המתאימה ולבסוף חוזר לקוד שנקטע.
  3. פעולות I/O והוראות מיוחסות רצות רק במצב גרעין. מצב גרעין יכול לקרוא וגם לכתוב את כל ה-RAM; מצב משתמש מוגבל לאזור שלו בלבד.
  4. קריאת מערכת עושה שני דברים: היא מעבירה את ה-CPU למצב גרעין וגם קופצת לקוד הגרעין (שגרת הטיפול בקריאת המערכת). ציון של החלפת המצב בלבד הוא חצי תשובה.
while(1){}    -> TIMER INTERRUPT (hardware) stops the runaway loop
interrupt     -> look up address in INTERRUPT VECTOR -> run the ISR
*null = 0     -> TRAP (software-generated, the process's own error)
write(fd,...) -> SYSTEM CALL: (1) switch to kernel mode (2) jump to kernel code
I/O / privileged instruction -> KERNEL MODE only (can access ALL of RAM)

  • פעולות I/O והוראות מיוחסות רצות רק במצב גרעין — לעולם לא "אחד משני המצבים". ומצב גרעין ניגש לכל ה-RAM, לא רק לאזור שלו.
  • כתובת ה-ISR נמצאת בוקטור הפסיקות (טבלת האיתור), לא סתם "אי שם בגרעין".
  • מלכודת = נוצרת בתוכנה (קריאת מערכת או שגיאה מקוד התהליך עצמו), ולא אירוע חומרה חיצוני.
  • פסיקת שעון = מכסת הזמן הסתיימה, והיא פסיקת חומרה — זה מה שמאפשר למערכת ההפעלה להפנות תהליך משתולל.
  • קריאת מערכת עושה שני דברים: מעבר למצב גרעין וגם קפיצה לקוד הגרעין. עצירה בהחלפת המצב מפספסת את העיקר.
  • פסיקות מקוננות (מודל הקורס): פסיקה חדשה שמגיעה בזמן ש-ISR רצה מטופלת אחרי שהנוכחית מסתיימת.
