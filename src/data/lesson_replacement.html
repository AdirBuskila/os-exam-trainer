Page Replacement, Thrashing & Working Set

When RAM is full and a page fault hits a page not in memory, the OS evicts a victim page. The exam asks for exact terms, the bit owners, and complexity of finding a victim. Answer in one or two words, in English.

Running a replacement algorithm on a reference string

  1. Draw a table: one row per access, one column per frame; start empty.
  2. For each reference: if the page is already in a frame → hit.
  3. If not → page fault; if a free frame exists, load it; else pick a victim by the rule.
  4. Victim rule: FIFO = oldest entry; LRU = used longest ago; OPT = used furthest in future; Clock = first R=0 (reset R=1 to 0 as you sweep); LFU = lowest counter.
  5. Count the page faults at the end.
Ref string: 1 2 3 4 1 2 5 1 2 3 4 5   |   3 frames   |   FIFO
Ref:  1  2  3  4  1  2  5  1  2  3  4  5
F1:   1  1  1  4  4  4  5  5  5  5  5  5
F2:   .  2  2  2  1  1  1  1  1  3  3  3
F3:   .  .  3  3  3  2  2  2  2  2  4  4
Flt:  F  F  F  F  F  F  F  H  H  F  F  H   ->  9 page faults

Working set rule: no thrashing while Σ|WSi| ≤ total frames. If demand D > memory → thrashing; a process without room for its working set is suspended. Note: paging moves one 4KB page; swap (roll out / roll in) moves an entire process.

Close-but-precise traps

  • Who sets which bit: the CPU/MMU sets the Reference and Dirty bits on access; the OS sets the Present bit and clears (resets) the Reference bit (e.g. in Clock). Reference/Dirty are never set by the OS.
  • Belady's anomaly is FIFO only. LRU and OPT are stack algorithms → immune. Don't blame LRU.
  • Clock complexity: worst case O(n) (all R=1 → full sweep), amortized O(1). The exam wants O(n).
  • OPT is not implementable — needs the future; it is only a benchmark.
  • LFU pitfall = "page stale": a once-popular page never gets evicted. (LRU does not have this.)
  • Reference vs LRU: Clock/second-chance approximates LRU cheaply; it is not exact LRU.
  • Global vs Local: global = victim from any process; local = victim only from the faulting process.
  • Thrashing spelling, and it means CPU is idle while disk thrashes — not "out of memory".
  • FIFO by entry order, LRU by use — do not mix the two when tracing.
