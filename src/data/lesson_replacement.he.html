Page Replacement, Thrashing & Working Set

כאשר ה-RAM מלא ומתרחש page fault על דף שאינו בזיכרון, מערכת ההפעלה מפנה דף קורבן (victim page). המבחן דורש מונחים מדויקים, מי הבעלים של כל סיבית, ואת הסיבוכיות של מציאת קורבן. ענו במילה אחת או שתיים, באנגלית.

הרצת אלגוריתם החלפה על מחרוזת ייחוס

  1. ציירו טבלה: שורה אחת לכל גישה, עמודה אחת לכל frame; התחילו ריקה.
  2. לכל ייחוס: אם הדף כבר נמצא ב-frame → hit.
  3. אם לא → page fault; אם קיים frame פנוי, טענו אותו; אחרת בחרו קורבן לפי הכלל.
  4. כלל הקורבן: FIFO = הכניסה הוותיקה ביותר; LRU = השימוש הכי מזמן; OPT = השימוש הכי רחוק בעתיד; Clock = ה-R=0 הראשון (אפסו R=1 ל-0 תוך כדי הסריקה); LFU = המונה הנמוך ביותר.
  5. ספרו את ה-page faults בסוף.

כלל קבוצת העבודה: אין thrashing כל עוד Σ|WSi| ≤ סך כל ה-frames. אם הביקוש D > הזיכרון → thrashing; תהליך שאין לו מקום לקבוצת העבודה שלו מושהה (suspended). שימו לב: דפדוף מעביר דף 4KB בודד; swap (roll out / roll in) מעביר תהליך שלם.

מלכודות קרובות-אך-מדויקות

  • מי קובע איזו סיבית: ה-CPU/MMU קובע את סיביות ה-Reference וה-Dirty בעת הגישה; מערכת ההפעלה קובעת את סיבית ה-Present ומאפסת (מנקה) את סיבית ה-Reference (למשל ב-Clock). סיביות ה-Reference/Dirty לעולם אינן נקבעות על ידי מערכת ההפעלה.
  • אנומליית Belady היא ב-FIFO בלבד. LRU ו-OPT הם אלגוריתמי מחסנית (stack algorithms) → חסינים. אל תאשימו את LRU.
  • סיבוכיות Clock: במקרה הגרוע O(n) (כל ה-R=1 → סריקה מלאה), משוערכת O(1). המבחן רוצה O(n).
  • OPT אינו ניתן למימוש — הוא דורש את העתיד; הוא רק קנה מידה (benchmark).
  • המלכודת של LFU = "page stale": דף שהיה פעם פופולרי לעולם אינו מפונה. (ב-LRU אין את הבעיה הזו.)
  • Reference מול LRU: Clock/second-chance מקרב את LRU בזול; הוא אינו LRU מדויק.
  • גלובלי מול מקומי: גלובלי = קורבן מכל תהליך; מקומי = קורבן רק מהתהליך שגרם ל-fault.
  • Thrashing איות, ומשמעותו שה-CPU בטל בעוד הדיסק עובד בתזזית — ולא "נגמר הזיכרון".
  • FIFO לפי סדר כניסה, LRU לפי שימוש — אל תערבבו בין השניים בעת מעקב.
