החלפת דפים ותזזית

תחת דפדוף לפי דרישה, ה-RAM מחזיק רק חלק מהדפים. כאשר page fault זקוק לדף חדש אך כל ה-frames מלאים, מערכת ההפעלה חייבת לבחור דף קורבן (victim page) לפינוי. אלגוריתמי החלפת הדפים הם פשוט הכללים השונים לבחירת אותו קורבן — ותזזית (thrashing) היא מה שקורה כשהמערכת טועה בגדול. ענו במילה אחת או שתיים, באנגלית.

איך עונים

  1. Optimal מפנה את הדף שייעשה בו שימוש הכי רחוק בעתיד; LRU מפנה את הדף שנעשה בו שימוש הכי מזמן בעבר. הניסוח "עתיד מול עבר" הוא הסימן המבחין.
  2. FIFO מפנה את הדף הוותיק ביותר, והוא האלגוריתם היחיד שסובל מאנומליית Belady (LRU ו-OPT הם אלגוריתמי מחסנית, חסינים).
  3. Clock: החומרה קובעת את סיבית ההתייחסות ל-1 בעת הגישה; מערכת ההפעלה מאפסת אותה ל-0 תוך כדי סריקה אחר קורבן. דף שהופנה אליו מקבל הזדמנות שנייה (הסיבית מאופסת והדף מדולג). מציאת קורבן היא O(N) במקרה הגרוע.
  4. הקצאה מקומית: תהליך שגרם ל-fault רשאי לפנות רק את הדפים שלו, ולכן הבאת דף עבור X מפנה דף של X.
  5. תזזית נובעת ממחסור ב-frames (תהליכים זקוקים ליותר frames מאשר קיימים). הפתרון הוא הורדת דרגת ריבוי התכנות (multiprogramming).
Clock sweep (hand looks for reference bit R = 0):
  page A  R=1  ->  second chance: OS clears R to 0, advance
  page B  R=1  ->  second chance: OS clears R to 0, advance
  page C  R=0  ->  VICTIM (evict C)
  worst case: hand visits all N frames  ->  O(N)

FIFO + Belady: more frames can give MORE faults (FIFO only).
Optimal: evict the page whose next use is farthest in the FUTURE.

מלכודות קרובות-אך-מדויקות

  • Optimal = עתיד, LRU = עבר. Optimal מפנה את הדף שייעשה בו שימוש הכי רחוק בעתיד; LRU מפנה את הדף שנעשה בו שימוש הכי מזמן. הניסוח "עתיד" מול "עבר" קובע את התשובה.
  • אנומליית Belady היא ב-FIFO בלבד. אם יותר frames גרמו ליותר faults, האלגוריתם הוא FIFO. LRU ו-OPT הם אלגוריתמי מחסנית ולעולם אינם סובלים מזה.
  • סיביות Clock: החומרה קובעת את סיבית ההתייחסות בעת הגישה; מערכת ההפעלה מאפסת אותה תוך כדי חיפוש הקורבן. דף שהופנה אליו מקבל הזדמנות שנייה (הסיבית מאופסת והדף מדולג), והחיפוש הוא O(N).
  • הקצאה מקומית מפנה דף של התהליך עצמו שגרם ל-fault; הקצאה גלובלית עשויה לפנות דף של כל תהליך.
  • תזזית = דרישה ליותר מדי frames ביחס ל-RAM הזמין, כך שמערכת ההפעלה מחליפה דפים ללא הרף. הפתרון הוא הורדת דרגת ריבוי התכנות, ולא שינוי האלגוריתם.
