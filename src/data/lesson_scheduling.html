CPU scheduling = the short-term scheduler picks which READY process gets the CPU. Know each algorithm's name, whether it is preemptive, and its one fatal flaw. For exam fill-ins, write the English term; the metric formulas and one Gantt chart cover the rest.

Short-term schedulerCPU schedulerמתזמן לטווח קצרPicks runnable→running; runs every few ms.
  1. List each process with Arrival and Burst (and Priority if given).
  2. Apply the algorithm rule at every event (a new arrival or a completion). SJF/Priority = pick best among already-arrived; SRTF/RR = re-check on each event/quantum.
  3. Draw the Gantt chart (time axis with who runs).
  4. Per process: Turnaround = Completion − Arrival; Waiting = Turnaround − Burst; Response = First-CPU − Arrival.
  5. Average = sum / n. Tie in burst → break by FCFS (arrival). Remember: smaller number = higher priority.

  • SJF is non-preemptive; SRTF is preemptive. Two different algorithms — do not mix.
  • RR with quantum = infinity (or Q ≥ longest burst) behaves like FCFS. Q too small → CPU wasted on context switches; Q too large → like FCFS.
  • Convoy effect belongs to FCFS (one long job blocks all). SJF/SRTF give the minimum average waiting time, but only on a given set.
  • Starvation is a consequence, not an algorithm. It hits SJF/SRTF (long jobs) and Priority (low priority); the fix is aging.
  • Waiting time uses burst, not total time. Waiting = Turnaround − Burst. Response = first time on CPU, not completion.
  • Priority: smaller number = higher priority in this course. nice range −20 (high) … +19 (low).
  • Linux = Multilevel Queue (FIFO real-time, RR real-time, non-real-time/epoch-based). One full cycle = Epoch; unfinished allocation carries half to the next epoch.
  • Windows: foreground-window thread gets a tripled quantum; after I/O, priority boost is inversely proportional to device speed (slow device → bigger boost), then decays each quantum.
  • RR is preempted by the timer interrupt (quantum end), never voluntarily.
