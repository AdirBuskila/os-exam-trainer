A process is a program in execution, represented by its PCB. Memorize the exact state names (standard 5 + Linux R/S/D/T/Z), what a context switch saves, and the zombie/orphan rule. Answers are one or two words — prefer the English term.

  1. State name asked → map the trigger: blocking syscall (read/wait/semop) → RUNNING→WAITING; quantum end / timer → RUNNING→READY; I/O done → WAITING→READY.
  2. Linux letter asked → disk read = D; wait-for-child / stdin = S; Ctrl+Z = T; finished-not-reaped = Z.
  3. "Where are registers saved on a context switch / blocking?" → always the PCB.
  4. zombie + orphan simultaneously → answer 0 (init adopts then immediately reaps).
  5. #processes after k forks → 2^k; getppid()==1 → parent died.
Process does read() from disk            -> state D (uninterruptible)
Parent does wait(), child still running  -> state S (sleeping)
Child exit(0), parent hasn't wait()ed    -> state Z (zombie)
Parent dies, child keeps running         -> orphan, getppid()==1
How many can be zombie AND orphan at once -> 0
  • Context switch ≠ mode switch. Mode switch = user↔kernel, same process (cheap). Context switch = swap whole process via PCB (expensive).
  • D vs S: D = uninterruptible (hardware/disk), cannot be woken by signal; S = interruptible (normal waiting).
  • R covers both Running and Ready in Linux — but they are formally two different states.
  • Zombie + orphan = 0, not 1. init always reaps; an orphan is adopted instantly.
  • Zombie still has a PCB (held only to deliver exit status); it owns no other resources.
  • exit status: 0 = success, any non-zero = failure. exit() flushes libc buffers; _exit() does not (use _exit in a forked child to avoid duplicate prints).
  • printf without \n + fork → buffer is copied → text prints twice. Fix with fflush or \n.
  • Registers on a blocking/context switch go to the PCB — not the stack, not RAM generically.
  • getppid()==1 means the parent finished (adopted by init), NOT that you are init.
