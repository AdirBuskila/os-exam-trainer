Processes & States

A process is a program in execution. The OS tracks many at once, so it gives each one a life-cycle of states (ready, running, waiting…) and a kernel record — the PCB that stores everything needed to pause and later resume it. This topic = those states, the transitions between them, and the PCB. Answers are one or two words — prefer the exact term.

Linux single-letter states:
  R = Running or Runnable (on a core, or busy-waiting / spinning)
  S = Interruptible sleep (I/O burst, full pipe, device request queue)
  D = Uninterruptible sleep (deep I/O wait, cannot be interrupted)
  Z = Zombie (finished, PCB not yet reaped)
  T = Stopped (suspended by a signal / debugger)
  1. Wants to run but there is no free CPU → Ready (not Waiting).
  2. Blocked on an I/O / event (I/O burst, full pipe, request queue) → S / Waiting; deep uninterruptible I/O → D.
  3. Busy-waiting (spinning) is using the CPU → R.
  4. Any per-process kernel data — saved registers, page-table address, nice value, owner — lives in the PCB.
  5. How many processes are zombie AND orphan at once → 0 (init adopts an orphan, then reaps it instantly).

  • Ready vs Waiting. "No free CPU" = Ready; "blocked on I/O / event" = Waiting / S. This distinction is a guaranteed point.
  • Busy-waiting is R (actively spinning on the CPU) — but an I/O burst, a full pipe, or a request queue is S.
  • Registers are saved in the PCB on a context switch — not on the stack, not "in RAM" generally.
  • Zombie + orphan = 0, not 1 — a zombie always still has a (live) parent.
  • init reaps orphans instantly: the moment a process is orphaned it is adopted by init (PID 1), which immediately reaps any child that dies.
