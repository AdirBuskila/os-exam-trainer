Memory & Paging: Virtual Memory, the MMU & the Page-Fault Path

Every program wants its own big, contiguous-looking block of memory starting at address 0, but real RAM is limited and shared by many processes. The fix is virtual memory + paging: give each program a fake (virtual) address space, chop it into fixed-size pages, and scatter those pages anywhere in physical RAM (or leave them on disk until needed). The MMU hardware translates the virtual address to the physical one on every single access.

Address & bit math — the reusable recipe

  1. offset bits = log2(page size) — the bits needed to address one byte inside a page.
  2. page-number bits = address bits − offset bits.
  3. number of page-table entries = 2 ^ (page-number bits) — one entry per virtual page.
  4. number of frames in RAM = physical memory size / page size = 2 ^ (physical bits − offset bits).
  5. TLB hit = 1 memory access (just the data). TLB miss with k-level paging = k + 1 accesses (walk all k tables, then the data).

Worked examples

2024A Q7 - 42-bit virtual address, 1MB pages:
  page size      = 1 MB = 2^20 bytes
  offset bits    = log2(2^20)      = 20
  #frames        = 2^(42 - 20)     = 2^22      <- answer

2021A Q8 - max internal fragmentation, page size 4096:
  max internal fragmentation = page size - 1
                             = 4096 - 1 = 4095 <- answer

3-level paging, on a TLB miss:
  accesses = 3 (one per page-table level) + 1 (the data)
           = 4                                 <- answer

The traps that cost points
  • The MMU is hardware that runs on every access — that is why it is hardware, not the OS (speed). It only signals the fault; the OS decides what to do (load from disk vs. kill the process).
  • The valid-invalid bit triggers the page fault. When it is off, the MMU raises a page fault and traps to the OS. Do not answer "dirty bit" or "reference bit" — those are for page replacement.
  • The TLB saves the EXTRA RAM access (the page-table read), not the data read. A k-level miss = k + 1 accesses; a 3-level miss = 4, not 3. The data still has to be read.
  • After a fault, re-run the faulting instruction — the very instruction that faulted, not the next one — and the page is loaded into RAM, not the TLB or cache.
  • Paging causes INTERNAL, not external, fragmentation — waste inside the last page (any frame fits any page, so no external fragmentation). Writing into your allocated-but-unused internal-fragmentation space → the value is written, no fault.
  • PTBR = 1 per core. A core runs one process at a time, so it needs exactly one "current page table" pointer.
