Interrupts & CPU Mode

This topic answers two needs of the CPU. First, it must react to events from the outside world (a key pressed, a disk finishing, the clock ticking) without constantly polling — that reaction mechanism is the interrupt. Second, it must protect itself so a buggy or malicious user program cannot touch hardware or other programs' memory — that protection mechanism is the split between kernel mode and user mode. Almost every exam card here is pure recall of the right term and who triggers it.

  1. Classify the trigger: a process error (div/0, NULL deref) → trap; an external hardware event (keyboard, disk, clock) → interrupt; an explicit request for an OS service → system call.
  2. When an interrupt fires, the CPU finds the handler's address in the interrupt vector, then jumps to the matching ISR and returns to the interrupted code afterwards.
  3. I/O and privileged instructions run only in kernel mode. Kernel mode can read and write all of RAM; user mode is fenced to its own region.
  4. A system call does two things: it switches the CPU to kernel mode and it jumps to kernel code (the syscall handler). Naming only the mode switch is half an answer.
while(1){}    -> TIMER INTERRUPT (hardware) stops the runaway loop

*null = 0     -> TRAP (software-generated, the process's own error)
write(fd,...) -> SYSTEM CALL: (1) switch to kernel mode (2) jump to kernel code
I/O / privileged instruction -> KERNEL MODE only (can access ALL of RAM)

  • I/O and privileged instructions run only in kernel mode — never "either mode." And kernel mode can access all of RAM, not just its own region.
  • The ISR address lives in the interrupt vector (the lookup table), not vaguely "somewhere in the kernel."
  • Trap = software-generated interrupt (a system call or an error from the process's own code), not an external hardware event.
  • Timer interrupt = the quantum expired, and it is a hardware interrupt — this is what lets the OS preempt a runaway process.
  • A system call does two things: switch to kernel mode and jump to kernel code. Stopping at the mode switch loses the point.
  • Nested interrupts (course model): a new interrupt arriving while an ISR is running is handled after the current one finishes.
