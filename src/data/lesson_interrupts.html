Interrupts, Traps & Mode Switching

The CPU normally runs user code in user mode; only the kernel can touch hardware. Three events flip it to kernel mode: a system call (process asks), an interrupt (hardware, async), or a trap (the process's own code errors, sync). Know the exact term and who triggers it.

  1. Read the trigger: a process error → trap; a hardware event → interrupt; an explicit request → system call.
  2. Any of the three causes a mode switch user→kernel (same process, cheap). Only the scheduler swapping processes is a context switch (saves/loads PCB, expensive).
  3. Hardware fires interrupt → CPU looks up the interrupt vector → jumps to the matching ISR → returns to interrupted code.
  4. Classify a call: needs the kernel/hardware → system call; pure user code → library call.
  5. I/O wait: spinning CPU = polling; device-to-RAM without CPU = DMA.
while(1){}   // runaway user process
  -> TIMER INTERRUPT (hardware PIT) fires at quantum end
  -> mode switch to kernel, run timer ISR (via interrupt vector)
  -> scheduler may CONTEXT SWITCH to another process
printf("hi")  -> library call (no mode switch)
write(fd,...) -> system call (mode switch into kernel)
*null = 0     -> TRAP (SIGSEGV, the process's own fault)
  • Trap vs Interrupt: trap is synchronous from the process's own code; interrupt is asynchronous from hardware. Don't swap them.
  • Mode switch ≠ context switch: mode switch = user↔kernel in the same process (cheap); context switch = swap processes, save/load PCB (expensive).
  • System call vs library call: printf/strcpy/malloc are library calls; fork/read/write/semop are system calls (mode switch). (malloc may internally call brk/mmap.)
  • DMA writes to RAM, not to the CPU — its whole point is to bypass the CPU.
  • Polling wastes CPU (busy asking); interrupts/DMA do not.
  • Who stops an infinite loop? The timer interrupt (generated by hardware — the PIT/timer), not the OS scheduler by itself.
  • Three (only three) ways into kernel mode: system call, interrupt, trap/exception. Nothing else.
  • Spool sits on the hard disk (HDD) — a print/job queue, not a fast cache.
