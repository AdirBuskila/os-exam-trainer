Synchronization: Race Conditions, Semaphores & the Classic Problems

The whole topic comes from one fact: when processes or threads share data and their steps interleave, a bad ordering can corrupt the data (a race condition). So the subject is really three things glued together: the problem (uncontrolled sharing), the tools that enforce taking turns (mainly semaphores), the classic problems you solve with them (producer-consumer, readers-writers, dining philosophers, bakery), and finally deadlock — what happens when locking itself goes wrong. The exam mixes pure theory with the exact behavior of the System V IPC syscalls.

System V IPC syscalls

SyscallBehavior the exam tests
semgetCreates or gets a set (array) of semaphores and returns a semid. It is what determines the array size the OS allocates — e.g. asking for 123 gives an array of 123 semaphores. IPC_PRIVATE makes a brand-new private set; the octal flag is permissions.
semopPerforms its operations atomically on the set. sem_op=-1 waits/decrements and the process sleeps/blocks if the value is already 0; sem_op=+1 signals/increments and wakes one waiter; sem_op=0 waits until the value becomes 0. With nsops>1 all operations succeed together or the whole call blocks (all-or-none).
semctlControl operations on a set. SETVAL/SETALL initialize the semaphore values; IPC_RMID deletes the whole set. So besides deleting, semctl is also what initializes values.
shmgetAllocates a shared segment of the requested size and returns a shmid. The size is rounded up to a multiple of the page size, so a shared region is always a whole number of pages.
shmat / shmdtshmat attaches the segment into the process address space and returns a pointer; shmdt detaches it. Calling shmdt on an address that is not an attached segment returns -1.
shmctlControl for shared memory; IPC_RMID frees / marks for deletion a segment created with shmget. This is how you free shared memory.
ipcrmShell tool that removes a leftover IPC object: -s a semaphore set, -m shared memory, -q a message queue.

Solving procedure — semop scenario

  1. Write down each semaphore's initial value (after SETVAL/SETALL).
  2. For sem_op=-1 (wait): if the value is >0 decrement it; if the value is 0 the process blocks (sleep/WAITING).
  3. For sem_op=+1 (signal): increment the value and wake one blocked waiter.
  4. For sem_op=0: continue if the value is 0, otherwise block until it reaches 0.
  5. If nsops>1: every operation in the call must succeed together (atomic, all-or-none) — e.g. grabbing two chopsticks at once.
Start: S=1
wait(S)  : sem_op=-1 → S=0, RUNNING
wait(S)  : sem_op=-1 → blocks (S was 0)
signal(S): sem_op=+1 → wakes one → RUNNING

What the exam asks

QuestionVerified answer
The three critical-section requirements?Mutual exclusion, progress, bounded waiting. Starvation is a consequence, not a requirement.
What is a race condition?A bug where the result depends on the interleaving order of shared access.
Binary vs counting semaphore?Binary 0/1 acts as a lock; counting 0..N counts available resource units.
What is TestAndSet?One atomic CPU instruction: returns the old value and sets the lock to true.
Why does the kernel use spinlocks across cores, not semaphores?A blocking semaphore wait sleeps, but the kernel cannot put itself to sleep.
Producer-Consumer semaphores and order?mutex=1, empty=N, full=0. Producer: wait(empty) then wait(mutex); Consumer: wait(full) then wait(mutex).
What if wait(mutex) comes before wait(empty)?Deadlock — the producer sleeps while still holding the lock.
Who starves in the default readers-writers?Writers (readers-preference); a writers-preference variant fixes it.
What is the bakery algorithm for?A software mutex for any number of processes, with no hardware and no OS calls.
When does bakery compare PIDs, and who wins?Only when two tickets are equal; the smaller PID/index enters first.
sem_op=-1 on a semaphore whose value is 0?The process sleeps/blocks — not an error, does not return -1.
Atomically wait on 5 and signal on 5?One semop call with nsops=10 sembuf entries — not ten calls.
semop on a set after IPC_RMID?Returns -1 (errno EIDRM).
Does filling sops.sem_op=-1 change the semaphore?No — nothing happens until you actually call semop.
Constraint on shmget size?Rounded up to a multiple of the page size.
ipcrm -s 666?Deletes semaphore set id 666.
The four Coffman conditions?Mutual exclusion, hold-and-wait, no preemption, circular wait — break any one to prevent deadlock.
Close-but-precise traps:
  • The three requirements are mutual exclusion, progress, bounded waiting — people forget bounded waiting. Starvation is a consequence, not a condition.
  • Producer-Consumer: always do wait(empty)/wait(full) before wait(mutex); reversing them deadlocks because you sleep holding the lock.
  • The default readers-writers solution starves writers (readers-preference).
  • The bakery algorithm compares PIDs only when tickets are equal; the smaller PID enters first.
  • sem_op=-1 on a 0-valued semaphore puts the process to sleep — not an error, and it does not return -1.
  • semctl also initializes values (SETVAL/SETALL), not just deletes (IPC_RMID).
  • A semop issued after the set was deleted with IPC_RMID returns -1.
  • Setting sops.sem_op without calling semop does nothing.
  • To atomically wait-on-5 and signal-on-5 you need one semop with nsops=10, not ten calls.
  • A shared-memory segment size is always a multiple of the page size; shmget rounds it up.
  • The kernel cannot sleep, so for cross-core mutual exclusion it uses spinlocks, not blocking semaphores.
  • Deadlock needs all four Coffman conditions (mutual exclusion, hold-and-wait, no preemption, circular wait); breaking one is enough to prevent it.
