Sync: Semaphores, Shared Memory & Critical Sections

Synchronization stops a race condition where the result depends on execution order. The exam loves exact terms and System V call behavior: sem_op=-1 is wait, +1 is signal, and you must know the 3 critical-section conditions cold. Write the English term even if you explain in Hebrew.

Solving procedure — semop scenario

  1. Write each semaphore's initial value (after SETVAL).
  2. For sem_op=-1: if value>0 decrement; if value=0 the process blocks (sleep/WAITING).
  3. For sem_op=+1: increment and wake one waiter.
  4. For sem_op=0: continue if value=0, else block until it reaches 0.
  5. If nsops>1: all ops succeed together or the whole call blocks (philosophers = 2 chopsticks at once).
Start: S=1
wait(S)  : sem_op=-1 → S=0, RUNNING
wait(S)  : sem_op=-1 → S=0, process SLEEPS (was 0)
signal(S): sem_op=+1 → S=0, waiter woken → RUNNING
Close-but-precise traps:
  • sem_op=-1 on a 0 semaphore → blocks/sleeps (not an error, not -1). 2024-B Q10 answer = sleep.
  • sem_op=-1 is wait/lock; +1 is signal/unlock — never reversed.
  • Setting sops.sem_op=-1 alone does nothing — it only acts when you call semop().
  • After IPC_RMID the array is gone; a later semop returns -1.
  • The 3 conditions are Mutual Exclusion, Progress, Bounded Waiting. Starvation is a consequence, NOT a condition.
  • Bakery without the choosing[] array breaks Mutual Exclusion (two pick same number, both enter).
  • Spinlock suffers from busy waiting + starvation; kernel uses it because it cannot sleep.
  • Producer-Consumer order matters: wait(empty) before wait(mutex), else deadlock.
  • semctl alone does SETVAL, GETVAL and IPC_RMID — same call, different cmd.
  • Shared memory has no built-in sync — always guard it with a semaphore. shmget size rounds up to 4K pages.
  • Coffman deadlock conditions (4): Mutual Exclusion, Hold-and-Wait, No Preemption, Circular Wait. Break one → no deadlock.
  • TestAndSet is one atomic CPU instruction (read old + set true); don't confuse with two separate steps.
  • Forgetting IPC_RMID / ipcrm leaves IPC objects until reboot — counted as wrong in code answers.
