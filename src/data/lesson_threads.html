Thread (lightweight process) — the basic unit of CPU utilization. A thread has its own stack and registers (plus a PCB in the kernel); it shares the code (text), data, and open files with its peer threads. That shared bundle is called a task. A traditional heavyweight process = a task with exactly one thread.

Why threads?

  • While one (server) thread is blocked/waiting, another thread in the same task can run.
  • Apps that share a common buffer (producer–consumer) benefit from threads.
  • They let sequential processes make blocking system calls while still achieving parallelism.

Types of threads

  • Kernel-supported threads.
  • User-level threads — supported above the kernel via a set of library calls.
  • Anderson's threads — like kernel threads, but send scheduling events up to the task instead of interpreting them alone.

Multithreading models

  • Many-to-One — many user threads → one kernel thread; used on systems with no kernel-thread support (e.g. CMU's Project Andrew).
  • One-to-One — each user thread → its own kernel thread (Windows 95/98, OS/2).
  • Many-to-Many — many user threads ↔ many kernel threads; hybrid of both (Solaris 2, Linux, Windows NT/2000).

Solaris 2

  • LWP (lightweight process) = the intermediate level between user-level and kernel-level threads.
  • User-level thread: needs only a stack + program counter, no kernel involvement → fast switching.
  • Kernel thread: small data structure + stack; switching is fast (no memory-map change). LWP switching is slower (a PCB with register/accounting/memory info).
